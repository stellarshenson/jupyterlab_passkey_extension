8. **Task - Rejected a passkey-management TUI** (v1.0.4): assessed and declined a proposed Textual TUI to manage issued/active passkeys<br>
   **Result**: Weighed adding a `/text-user-interface`-style TUI to manage "issued / remembered / active" passkeys and concluded it does not fit this extension. The extension is a stateless bridge that performs no cryptography and holds no credential registry - passkeys live in the OS or browser authenticator (Windows Hello, security keys) and are referenced only by a caller-supplied `cred_id`, while policy and lifecycle state belong to the consuming vault or CLI. The only local state is the transient one-shot `/dev/shm/jlab-passkey-<uid>/<nonce>.json` relay, consumed within seconds, so a TUI here would have no real registry to manage and would merely duplicate the OS authenticator's own passkey UI. Building it would pull policy into a purpose-agnostic component, breaking the stateless-bridge design and the No-Slop rule; a management surface belongs in the caller. No code changed - decision only, and the Star Colonel agreed to skip it.

16. **Task - Leading-dash base64url broke the passkey CLI** (v1.0.12): `pass-cli-enroll --passkey` died at `get` with "expected one argument" immediately after a register ceremony the user had approved<br>
    **Result**: base64url's alphabet includes `-`, so ~1 in 64 cred_ids and PRF salts begin with one, and argparse reads any leading-dash token as an option - the documented `--cred-id "$cred"` therefore dies before any ceremony runs. Deterministic per credential, not flaky, which is how it passed a release. Evidence came from the real run's transcript in another session rather than the symptom: a discriminating test proved `--cred-id ""` is *accepted* while a leading dash reproduces the exact error, killing the "empty cred_id" theory. The suite could never catch it - every test calls `cmd_get(_ns(...))` directly, never argv. Added `_glue_b64url` to `cli.py` rewriting `--flag VALUE` to `--flag=VALUE`, with three tests driving `main()`. Same trap fixed in the vault's `pass-cli-keystore.py` - `add-passkey`, and `get-passkey`, the unlock path, where an unlucky passkey would enroll then never open the vault. pytest 65.
